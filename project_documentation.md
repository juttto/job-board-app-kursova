# Технічна документація: Веб-додаток "Біржа праці" (Job Board)

## 1. Вступ та Актуальність Проєкту

Сучасний ринок праці вимагає гнучких, швидких та інтуїтивно зрозумілих інструментів для взаємодії між роботодавцями та шукачами роботи. Цей веб-додаток, розроблений як курсовий проєкт, є повноцінною імітацією сучасної "Біржі праці" (Job Board).
Він вирішує дві ключові проблеми:
1.  Спрощення процесу публікації вакансій для компаній-роботодавців.
2.  Надання кандидатам зручного інструментарію для пошуку, сортування, збереження вакансій та швидкого відгуку на них за допомогою персонального цифрового резюме.

Цей документ є вичерпним довідником щодо внутрішньої організації коду, архітектурних патернів, структури даних та логіки взаємодії компонентів.

---

## 2. Технологічний Стек та Архітектурний Патерн

Проєкт побудований на архітектурі **SPA (Single Page Application)** з використанням парадигми інкапсульованих компонентів та безсерверної (Serverless-like) моделі роботи з даними.

### 2.1. Основні технології
*   **React 19:** Бібліотека для побудови користувацьких інтерфейсів на основі компонентного підходу. Використовуються функціональні компоненти та хуки (Hooks: `useState`, `useEffect`, `useContext`, `useCallback`).
*   **Next.js 15:** Мета-фреймворк поверх React. Забезпечує:
    *   **App Router:** Нова система маршрутизації на основі файлової структури (папки відповідають за URL).
    *   **Оптимізація:** Автоматична мініфікація бандлів та оптимізація зображень (хоча в проєкті переважає клієнтський рендеринг `use client`).
*   **Tailwind CSS:** Utility-first CSS фреймворк, що дозволив уникнути створення десятків CSS-файлів. Дизайн створюється безпосередньо в JSX-розмітці за допомогою класів на кшталт `flex flex-col items-center justify-between p-24`.
*   **Next-Themes:** Бібліотека для реалізації "Темної / Світлої" теми. Працює шляхом динамічної підміни CSS-змінних на кореневому елементі `<html>`.
*   **Lucide-React:** Бібліотека векторних іконок, що гарантує єдиний візуальний стиль усіх піктограм.
*   **Local Storage API (Mock DB):** Оскільки проєкт автономний і не має підключеного Node.js / Python сервера, вся робота бази даних (`SELECT`, `INSERT`, `UPDATE`, `DELETE`) імітується через взаємодію з пам'яттю браузера.

## 3. Структура Директорій (App Router File-System Based Routing)

Директорія `src/app/` є ядром маршрутизації. Кожна папка з файлом `page.js` стає публічним маршрутом (URL).

```text
job-board/

├── src/
│   ├── app/                # Система маршрутизації (Next.js App Router)
│   │   ├── globals.css     # CSS-змінні для Tailwind (Colors, Radiuses, Theme)
│   │   ├── layout.js       # RootLayout: Обгортає всі сторінки провайдерами (Contexts)
│   │   ├── page.js         # СТОРІНКА: "/" - Головна (Hero Section, Статистика, Пошук)
│   │   ├── candidates/     # СТОРІНКА: "/candidates" - Довідник усіх резюме
│   │   ├── companies/      # СТОРІНКА: "/companies" - Довідник компаній
│   │   ├── create-resume/  # СТОРІНКА: "/create-resume" - Форма кандидата
│   │   ├── jobs/
│   │   │   ├── page.js     # СТОРІНКА: "/jobs" - Каталог усіх вакансій + Фільтри
│   │   │   └── [id]/       # СТОРІНКА: "/jobs/:id" - Динамічний маршрут (Деталі)
│   │   ├── login/          # СТОРІНКА: "/login" - Вхід
│   │   ├── post-job/       # СТОРІНКА: "/post-job" - Форма роботодавця
│   │   ├── profile/        # СТОРІНКА: "/profile" - Особистий кабінет (Dashboard)
│   │   └── register/       # СТОРІНКА: "/register" - Реєстрація (Вибір ролі)
│   ├── components/         # Reusable Components (Модулі інтерфейсу)
│   │   ├── Footer.js       # Підвал сайту

│   │   └── ThemeToggle.js  # Кнопка Сонце/Місяць

│   │   └── ToastContext.js # Система спливаючих повідомлень

│   │   └── jobs.json       # Хардкод база вакансій для стартового наповнення
│   └── utils/
│       └── api.js          # CRUD операції для LocalStorage (імітація Backend)
```

## 4. Глобальні Стани (Context API) та Управління Даними

### 4.1. AuthContext: Авторизація та Збереження Сесії
Традиційна проблема React-додатків — втрата стану при перезавантаженні сторінки (F5). Для її вирішення створено `AuthContext`. 

**Концепція "Hydration":**
При монтуванні Root Layout спрацьовує `useEffect` в `AuthContext`, який зазирає в `localStorage`, шукає ключ `jobboard_user` і відновлює сесію. Стан `loading` блокує рендер захищених сторінок, поки ця перевірка не завершиться.

```javascript
// Скорочений приклад роботи AuthContext
export const AuthProvider = ({ children }) => {
    const [user, setUser] = useState(null);
    const [loading, setLoading] = useState(true);

    useEffect(() => {
        const storedUser = localStorage.getItem("jobboard_user");
        if (storedUser) setUser(JSON.parse(storedUser));
        setLoading(false); // Глобально дозволяємо рендер після перевірки
    }, []);

    };
```

### 4.2. ToastContext: Система Візуального Сповіщення (Feedback)
Для покращення UX розроблена система `Toast` — спливаючих вікон, що сигналізують про успіх або помилку (напр.: "Резюме успішно збережено").
Замість важких npm-пакетів реалізовано власне рішення на 100 рядків коду:
*   `ToastContext` зберігає масив активних сповіщень.
*   Функція `showToast` генерує ID (`Date.now()`), додає об'єкт в масив і встановлює `setTimeout(..., 4000)` на його автоматичне видалення.
*   Кастомний компонент `ToastItem` використовує inline CSS-transitions (`opacity 0.3s, transform 0.3s`) для плавної анімації виїзду за межі екрану.

## 5. Модель Даних (Mock Database) та Типізація

### 5.1. Патерни взаємодії з LocalStorage (`utils/api.js`)
Уся логіка "бази даних" винесена у файл `api.js` за допомогою асинхронних функцій-заглушок (`async/await`). Це дозволяє компонентам робити запити типу `const data = await getJobs()`, абсолютно не знаючи, що дані беруться з браузера, а не з сервера.

**Проблема об'єднання даних (Merkle / Merging):**
Платформа завжди показує суміш із "захардкоджених" вакансій (`data/jobs.json`) та свіжих локально створених вакансій.
Алгоритм `getJobs()` спочатку читає масив `jobboard_local_jobs`, потім імпортує `jobs.json`, і повертає злитий масив `[...localJobs, ...staticJobs]`.

**Проблема конфлікту типів (Type Coercion):**
Динамічні роути Next.js (як `/jobs/123`) завжди передають ID у вигляді `String` (`"123"`). У цей же час, дані у JSON файлі містять ID як `Number` (`123`). Використання жорсткого порівняння `===` приводило до помилок "Вакансію не знайдено". 
**Рішення:** Глобальне використання експліцитного приведення до рядка перед будь-яким порівнянням ідентифікаторів: `String(job.id) === String(url_id)`.

### 5.2. Структура Ключів в Local Storage
*   `jobboard_users_list`: `[{ id, email, password, role }]` — реєстр усіх користувачів.
*   `jobboard_user`: `{ id, email, role }` — активний токен (сесія).
*   `jobboard_local_jobs`: `[{ id, title, company, ... employerId }]` — вакансії користувачів.
*   `jobboard_saved_jobs_{UserID}`: `["jobId1", "jobId2"]` — обране для кандидатів.
*   `jobboard_applications_{JobID}`: `[{ candidateId, name, email, date }]` — відгуки.
*   `jobboard_resume_{UserID}`: `{ skills, experience, education }` — резюме кандидата.

## 6. Рольова Модель та Логіка Контролю Доступу (Role-Based Access Control)

Проєкт використовує жорсткий поділ функціоналу (Role: `employer` або `candidate`). Це реалізується двома способами:

### 6.1. Захист Сторінок (Route Guards)
Якщо користувач не авторизований, або намагається зайти на сторінку іншої ролі, його зупиняє компонент на рівні рендеру сторінки.

**Приклад доступу до `/post-job` (Створення вакансії):**
1. Читаємо `AuthContext`.
2. Якщо `loading === true`, показуємо індикатор звантаження (Spinner).
3. Якщо `!user`, виконується `router.push("/login")`.
4. Якщо `user.role !== "employer"`, рендеримо: `<h1>Доступ заборонено. Тільки для роботодавців.</h1>`.
5. Інакше — відмальовуємо форму.

### 6.2. Умовний Рендеринг Компонентів (Conditional UI)
Кнопки взаємодії на сторінці `/jobs/[id]` динамічно підлаштовуються:
*   Якщо ти — Роботодавець, який створив цю вакансію: ховаються всі кнопки, щоб ти не відгукнувся сам на себе.
*   Якщо ти Кандидат — додаються перевірки:
    *   Чи є ID цієї вакансії у масиві `saved_jobs`? (Змінити кнопку на "Збережено").
    *   Чи є ID кандидата у масиві `applications` для цієї вакансії? (Змінити кнопку на "Ви відгукнулись" і зробити її сірою / `disabled`).

## 7. Детальний Розбір Життєвого Циклу Сторінок

### 7.1. Особистий Кабінет (`/profile`) Dashboard
Це найскладніша сторінка проєкту, оскільки її інтерфейс на 100% поліморфний (залежить від ролі). Сторінка має "Вкладки" (Tabs), реалізовані через локальний стан `useState("metrics")`.

**Логіка Роботодавця у Профілі:**
1.  **Tab "Мої вакансії":** Викликає `getJobs()`, і фільтрує результат суворо за `employerId === user.uid`. У списку до кожної картки вакансії прикріплюється кнопка **Видалити**. Клік на неї викликає `deleteLocalJob(id)` та оновлює стан компонента без перезавантаження сторінки.
2.  **Tab "Відгуки":** Проходить по всіх своїх вакансіях, перевіряє ключі `jobboard_applications_{id}` і формує консолідований список: Назва вакансії -> Кандидат, що відгукнувся -> Його E-mail.

**Логіка Кандидата у Профілі:**
1.  **Tab "Резюме":** Перевіряє наявність ключа `jobboard_resume_{UID}`. Якщо даних немає, попереджає про необхідність створення резюме (з кнопкою переходу на `/create-resume`).
2.  **Tab "Збережені":** Читає масив збережених ID, запитує всі вакансії `getJobs()`, і фільтрує лише ті об'єкти, чий `id` є у масиві збережених.

### 7.2. Динамічні Каталоги (`/companies` та `/candidates`)
Щоб уникнути створення окремих складних сутностей для Компаній, у проєкті використано алгоритм динамічної дедуплікації (Dynamic Computed Directories).

Для сторінки `/companies` додаток не звертається до списку користувачів. Він завантажує ВСІ вакансії платформи і програмує цикл `reduce()`, який групує вакансії за полем `job.company`.
Якщо компанія зустрічається вперше, для неї створюється об'єкт. Якщо вдруге — їй додається +1 в поле `activeJobs`. У результаті на екран виводяться красиві картки організацій з реальною цифрою відкритих вакансій, зібраних у реальному часі.

### 7.3. Конструктор Форм та Валідація
Форми створення вакансії та резюме використовують контрольовані компоненти React (Controlled Inputs).
*   Наприклад, форма резюме має стан `formData = { skills: "", education: "", experience: "" }`.
*   Поля `required` контролюються вбудованою HTML5-валідацією перед викликом `onSubmit`.
*   Локація і спеціалізація збираються як текстові змінні, що дозволяє виконувати пошук `indexOf` або `toLowerCase().includes()` на головній сторінці.

## 8. Дизайн Засоби (Design System) та Інтерфейс

Вся презентаційна частина побудована на системі дизайн-токенів Tailwind CSS:
1.  **CSS Змінні:** Кольори не жорстко зашиті. Вони прив'язані до змінних, наприклад `--background: 0 0% 100%;` (світло), і `--background: 222.2 84% 4.9%;` (темно). При підключенні плагіну `next-themes` ці змінні реактивно змінюються, забезпечуючи перемикання теми (Dark Mode).
2.  **Глассморфізм (Glassmorphism):** Навігаційне меню має клас `backdrop-blur-md bg-background/80`, що створює ефект прозорого скла при скролі сторінки.
3.  **Анімації (Micro-interactions):** Кожна картка та кнопка мають перехідні стани, які активуються при наведенні (Hover). Наприклад, клас `transition-all duration-300 hover:scale-[1.02] hover:shadow-lg` візуально "піднімає" картку вакансії, коли користувач наводить на неї курсор.

## 9. Висновок

Даний веб-додаток "Біржа праці" є комплексним багатосторінковим вирішенням (SPA routing pattern), що демонструє всі ключові навички сучасної фронтенд-розробки. Використання Context API, розумне конструювання Mock-бази даних на LocalStorage з вирішенням проблем типізації, а також застосування патернів захисту маршрутів (RBAC) роблять його надійним прототипом для впровадження на реальних бекенд-системах у майбутньому.
